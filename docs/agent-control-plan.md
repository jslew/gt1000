# GT-1000 Agent Control Plan

## Goal

Let an agent inspect and edit a connected GT-1000 from natural-language requests while keeping MIDI writes validated, reversible where practical, and grounded in decoded device state.

## Milestones

1. Stabilize the protocol layer
   - Continue moving raw SysEx construction into typed enums and builders.
   - Model addresses, targets, value ranges, Assign fields, effects, chain elements, and parameter encodings explicitly.
   - Keep exact byte-level tests for every protocol builder.

2. Read the current patch
   - Add RQ1 reads for the temporary patch data needed to understand the active patch.
   - Start with patch name, BPM, master settings, chain layout, enabled blocks, effect types, key parameters, and assigns.

3. Decode a patch snapshot
   - Build a Python patch snapshot model from raw SysEx replies.
   - Represent patch name, BPM, chain elements, effects, assigns, master settings, and routing as structured Python data.

4. Describe the signal chain
   - Generate a readable signal-chain summary from a patch snapshot.
   - Include block order, on/off state, effect types, key parameters, split/merge routing, and output routing.

5. Add safe edit commands
   - Define validated commands such as `setEffectEnabled`, `setParameter`, `setBPM`, `setAssign`, `moveChainElement`, and `changeEffectType`.
   - Validate addresses, ranges, encodings, firmware quirks, and grouped-parameter requirements before sending MIDI.

6. Add an intent layer
   - Map natural-language-style intents to safe edit commands.
   - Examples:
     - "Make the delay darker" lowers delay high-cut or tone.
     - "Add slapback" enables delay and sets short time, low feedback, and conservative level.
     - "Make it more ambient" adjusts delay/reverb mix within bounded ranges.
     - "Turn on tuner" installs the tuner Assign and sends CC#80.

7. Add an agent interface
   - The agent should inspect decoded snapshots and propose structured commands.
   - The Python CLI should execute only validated commands, not arbitrary raw SysEx generated by the agent.
   - Show a command log and, for risky edits, a before/after diff before applying.

8. Add richer CLI support
   - First keep read-only patch inspector views reliable.
   - Then add signal-chain summaries, control maps, and command logs.
   - Later add an intent interface backed by structured commands.

9. Verify continuously
   - Keep Python unit tests for encoding and decoding.
   - Keep live verification commands for read patch, describe chain, dry-run edit, apply edit, and re-read patch to confirm the device state.

## Feature Implementation Status

Status legend:

- `Complete`: implemented as a supported CLI workflow.
- `Partial`: implemented for a useful subset, but not Tone Studio parity.
- `Planned`: not implemented as a first-class workflow yet.

### Near-Term Priority

Focus implementation on the first three P1 rows:

1. Expand full effect parameter schemas and editors.
2. Add general CTL/EXP, Assign, Patch MIDI, and LED editing.
3. Add library and patch-management operations.

These unlock the highest-value Tone Studio editing workflows while preserving the repo's core safety rule: writes must go through typed, validated commands with read-back verification where practical.

## Current MIDI Findings To Preserve

- GT-1000/GT-1000CORE SysEx model ID is `00 00 00 4F`.
- Roland/BOSS DT1/RQ1 checksums are calculated over address plus data or size only.
- GT-1000 BPM values are encoded as four 4-bit nibbles of `BPM * 10`.
- Tested GT-1000 v4 with Bass Mode off responds to Assign target `987` for `TUNER ON/OFF`.
- CC#80 is encoded as Assign source byte `0x45`, while the outbound MIDI controller byte is `0x50`.
- Assign target min/max values use a `32768` offset.
- MIDI CC active range should match `0...127`.
